
## 3. ユーザーストーリー

### 概要
ユーザーの視点から具体的な利用シナリオを記述し、開発における優先度と受け入れ基準を明確化するドキュメントです。プロジェクト憲章で定めた「提供価値」を、ユーザーの具体的な行動と成果に落とし込みます。

**必要性:**
- なくてもプロジェクトは進められる
- 要件の漏れ防止やユーザー目線での利便性の追求（提供価値の達成）のためにはあった方がよい
- 要求仕様書の機能要件を導出する際の重要な参考資料となる

**作成タイミング:**
- プロジェクト憲章の後、要求仕様書の前
- 要求仕様書作成と並行して詳細化することも可能

### 記載項目

#### 3.0 更新履歴
- **記載内容:** このドキュメントをだれがいつなにを更新したかを記述(CHANGELOG相当)
  - 日付
  - 担当者
  - 変更内容
- ドキュメントのリリースごとに更新

#### 3.1 ユーザーペルソナ
- **記載内容:** システムを利用する代表的なユーザー像
- 各ペルソナについて以下を記述:
  - **ペルソナID:** 識別子（例: P-001）
  - **名前/種別:** ペルソナの名称（例: 「管理者」「一般ユーザー」「ゲスト」）
  - **属性:** 年齢、職業、技術スキルレベルなど
  - **背景:** このシステムを利用する前の状況
  - **目標（ニーズ）:** このユーザーが達成したいこと
  - **課題（ペインポイント）:** 現在抱えている問題や不満
  - **利用シーン:** いつ、どこで、なぜシステムを使うのか
  - **期待すること:** システムに対する期待

#### 3.2 ユーザーストーリー
- **記載内容:** ペルソナごとの具体的な利用シナリオ
- 各ストーリーに以下を含める:
  - **ストーリーID:** 識別子（例: US-001）
  - **優先度:** 高/中/低（またはMust/Should/Could）、もしくはもっと汎用な優先順位（1~n）
  - **As a（誰が）:** どのユーザーペルソナ種別として
  - **I want to（何をしたいのか）:** 具体的な行動や機能
  - **So that（なぜそれをしたいのか）:** その結果得られる価値や達成できること
  - **受け入れ基準:** 各ユーザーストーリーが「完了」とみなされる条件
    - テスト可能な形で記述する
    - この受け入れ基準は、後のテスト仕様書作成の基礎となる
